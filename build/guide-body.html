What students see

A worksheet page has three panels: exercise list (left), the current derivation tree (centre), lexicon and symbol reference (right). The workflow per derivation: click a node whose children are resolved, choose a composition rule, type the β-reduced result, submit. The grader accepts any answer α-equivalent to the target after normalization; conjunct order does not matter. Choosing an inapplicable rule is refused with the reason (for example: “Both sisters must be predicates ⟨A,t⟩ of the same type A. The sisters are ⟨e,⟨e,t⟩⟩ and e.”).

Input accepts ASCII shortcuts: Lx. → λx., & → ∧, ~ → ¬, -> → →, and a clickable symbol palette covers the rest. The palette shows only the operator symbols the current worksheet uses (a §6 worksheet shows no mereology or presupposition keys; the §10 worksheets show ∗ ⊕ ≤; the definedness and tense/modal worksheets show ∂ ⊤ ⊥).

On first opening a worksheet, a card lists the composition rules enabled in it, with a description and a usage note per rule. It appears once per worksheet per browser; the ☰ Rules button reopens it.

Where the author provided them, derivations carry staged hints (revealed one at a time; in practice mode the final stage offers “Show answer”) and worksheets carry a notes panel — a reading companion with numbered examples, trees and denotation tables. Exercises can anchor to sections of the notes; opening an exercise scrolls the notes to its section.

Progress is stored in the browser's localStorage, separately per area: the /cc pages (whole-book and chapters) share one record; /hk and /papers each have their own; each hosted version has its own. Tools menu → “Save progress to a file” writes all of it to a JSON file; “Restore progress from a file” loads it on another machine. A completion summary (per-worksheet solved counts) is available from the same menu. The curated pages register a service worker on first visit and afterwards load without a network connection.

Authoring worksheets

The editor at /editor runs without an account. A worksheet consists of: a domain (constants and variables with types), a lexicon (words with λ-denotations), a rule configuration, and exercises (groups of derivations, each a bracketed tree with optional targets, hints and notes anchors). The file format is JSON, documented in FORMAT.md; any worksheet from /files can be used as a template.

Opening and importing

Click the worksheet name in the header to open the Worksheets panel. + New creates an empty worksheet; dropping a .compose.json or .compose-bundle.json onto the panel imports it. Imported and newly authored worksheets are stored in the browser.

The editor surface

Left column: checkboxes for the five composition rules (FA, PM, NN, PA, IFA), the type-shifting inventory grouped by family (NP shifts, adjectives, closure, argument raising), behaviour switches (Quantifier Raising, auto-resolve non-branching nodes), and domain operators (∗ ⊕ ≤ ∂ ⊤ ⊥) — students' symbol palettes show exactly the operators enabled here. Centre: the lexicon. Words are comma-separated for synonyms (runs,run); denotations are typed in ASCII (Lx.Ly.greet(y,x)). The engine parses each entry as you type and displays its computed type, or a parse error. Below the lexicon: constant declarations (f g : e) and variable-type declarations, with one-click additions for events, worlds, times, generalized quantifiers.

Derivations

Right column: exercises. A derivation is a labelled-bracket tree — [.S [.DP Frodo ] [.VP runs ] ] — where leaves are lexicon words. Optional fields per derivation: a display sentence, an expected root denotation, a teacher-only note, and hints (one per line, revealed in order). ▶ Test solves the tree with the engine and shows the computed root meaning and type; the expected field shows ✓ if they match, ✕ if not. Open first tree → loads the worksheet in the app as a student would see it.

Notes

The 📝 Notes button opens a two-pane editor: source on the left, rendered student preview on the right. The input format is Markdown for document structure with LaTeX for linguistic content — the full construct list is in the reference below. ## headings become sections; a derivation's “section” field anchors it to one, and the student notes panel scrolls there when the derivation is opened.

Saving and exporting

Header buttons: Save to library (stores in this browser), Load into app, .json (downloads the worksheet source), .html (downloads a self-contained student page). The HTML export — also available for multi-worksheet selections via Tools → Export assignment — is a single file with the app and the chosen worksheets embedded; it opens from disk without a network connection and is fixed in student mode. The JSON export is the re-importable source, and is also what you upload to a hosted version later.

Notes input reference

Notes (worksheet reading companions and version notes) are written in Markdown with a fixed set of LaTeX constructs. The renderer implements the constructs below natively; commands outside this list are shown as literal source rather than dropped.

Glosses (expex \begingl)

Input:

\begingl
\gla neko-ga neru //
\glb cat-NOM sleep //
\glft 'the cat sleeps' //
\endgl

Result: word-aligned two-line gloss with the translation beneath. Leipzig abbreviations in the gloss line (NOM, 3SG, NEG, …) are set in small caps and show their expansion on hover. A \begingl block inside \ex…\xe attaches the gloss to that example's number.

Trees (qtree, forest)

Input, either form:

\Tree [.S [.NP Frodo ] [.VP runs ] ]

\begin{forest}
[S{run(f)} [DP{f} Frodo] [VP{Lx.run(x)} runs]]
\end{forest}

Result: a typeset syntax tree. {…} after a node label is a denotation, displayed under the category label. Wrapping a terminal in <…> draws a roof. Forest option preambles before the first bracket are ignored.

Denotation tables (COMPOSE-specific)

LaTeX has no standard environment for type-annotated denotation steps, so this one construct is COMPOSE's own. One step per line, expression : type; the types right-align into a column; indentation indents the row; blank lines leave a gap.

\begin{derivation}
[[runs]]       = Lx.run(x) : <e,t>
[[Frodo runs]] = run(f)    : t
\end{derivation}

There is also an avm environment for feature structures (ATTR: value per line, nesting with brackets, & and \\ row syntax accepted).

IPA (tipa subset)

\textipa{…} maps a subset of tipa ASCII to IPA: @→ə E→ɛ I→ɪ O→ɔ U→ʊ V→ʌ &→æ S→ʃ Z→ʒ T→θ D→ð N→ŋ ?→ʔ :→ː "→ˈ %→ˌ, among others. Unmapped characters pass through unchanged.

Hosting for a class

Accounts

Registration at /dash requires an email address, a password of at least 10 characters, and an invite code (available from the administrator — see /about). No confirmation email is sent and there is no self-service password reset; a forgotten password is reset by the administrator.

Versions

A version is a hosted worksheet collection with its own student URL. + New version creates one from a title; the server assigns the slug. The version card shows: the /v/… link, a copy button, the worksheet count, a live indicator (published; unpublishing makes the student link return 404), ✎ Open editor, ▤ Share, 📖 Notes, ⬇ bundle (downloads the version's worksheets as a .compose-bundle.json), ⬆ import (replaces the content from a bundle file), delete, and an opens counter that increments each time the student page is loaded.

Content gets into a version three ways: import a bundle file, save a worksheet from the version's editor (✎ Open editor opens the editor with the version's own worksheets only; ☁ Save to server writes back), or author in the sandbox first and import the JSON. Every save is validated server-side: the same engine parses every worksheet and rejects the save with exact error paths (for example worksheets[0].content.lexicon[2].denotation) if anything fails; limits are 2 MB, 40 worksheets, 400 derivations per version, unique worksheet keys.

Sharing

▤ Share shows the student link as text, a QR code for it, a PNG download of the QR, and a printable A4 handout (title, QR, link). Edits to a published version take effect at the same URL — students see updated content on their next page load, and printed QR codes remain valid.

What the student link opens

/v/<slug> serves the same app as the curated pages, loaded with the version's worksheets. Progress on a version is stored under its slug, separately from the library pages and other versions. If the version has notes, students get the 📝 panel; the rules card and hints work as on library pages.

Files & offline use

Offline behaviour: the curated pages and the root work offline after a first visit (service worker; /dash and /edit are never cached). For fully offline distribution, the editor's HTML export produces single files that run from local disk with no server involved.
